# AI Game Studio - 系统总架构文档

## 1. 项目定义
本项目是一个“本地可视化控制台 + 云端 Cursor Cloud Agents 执行”的多角色游戏开发协作系统。目标不是做一个纯展示 UI，而是让多个角色化 Agent 按职责协作，逐阶段完成一个点击成长类 Web 小游戏，并把全过程可视化。

## 2. 第一版范围
第一版角色：
- 制作人 Producer
- 策划 Designer
- 程序 Developer
- 美术 Artist
- 测试 QA

第一版目标：
- 完成一个点击成长类 Web 小游戏
- 验证多角色协作是否成立
- 验证任务拆解、阶段推进、验收闭环是否成立
- 验证真实代码产出是否可落地

## 3. 系统分层
### 3.1 前端控制台
负责：
- 办公室场景展示
- 角色状态展示
- 对话气泡展示
- 小黑板任务板
- 留言板/日志面板
- 用户命令输入
- 阶段验收按钮
- 电脑占用与队列可视化

### 3.2 本地后端调度核心
负责：
- 全局状态管理
- 角色状态机
- 阶段推进
- 电脑锁与队列
- 调用 Cursor Cloud Agents API
- 接收 webhook
- 对话消息归档
- 任务计划与验收管理
- 将 agent 输出转为前端可消费状态

### 3.3 云端 Cursor Cloud Agents
负责：
- 需求理解
- 阶段计划生成
- 玩法设计
- UI 视觉规范设计
- 代码实现
- 测试与反馈
- 命令执行
- 文档产出

### 3.4 项目仓库
负责：
- 存放小游戏代码
- 被 Developer 在获得电脑锁后执行修改
- 保存 README、设计文档、测试报告等项目资产

## 4. 核心设计原则
1. 角色职责边界清晰
2. 只有一个角色可同时获得“电脑写入权限”
3. 阶段推进必须经过制作人与用户确认
4. 所有关键信息都需要落盘保存
5. 优先结构化输出，便于调度
6. 第一版先做单项目、单电脑、单主流程，不做并行仓库

## 5. 三个核心视图
### 5.1 空间视图
办公室、角色、电脑、排队区、黑板、留言板。

### 5.2 流程视图
当前阶段、待办、进行中、验收中、已完成。

### 5.3 协作视图
谁与谁沟通、谁把任务交给谁、当前依赖关系。

## 6. 单电脑机制
办公室只有一台电脑。只有获得电脑锁的角色可以：
- 修改代码
- 创建/删除文件
- 执行命令行
- 运行项目
- 修复缺陷

其他角色可以：
- 沟通
- 规划
- 输出文档
- 复核
- 测试方案编写
- 申请进入队列

## 7. 第一版目标小游戏
固定为：
- 用户点击主按钮获取资源
- 资源可用于升级点击收益
- 页面显示资源数、等级、升级按钮、反馈效果
- 有基础 UI 风格
- 有最小可运行交互闭环

## 8. 成功标准
1. 用户能将需求交给制作人
2. 制作人能产出阶段计划
3. 角色能按计划协作
4. Developer 能在电脑锁下完成真实代码修改
5. QA 能做阶段验收
6. 最终产出一个可运行点击成长 Web 小游戏
